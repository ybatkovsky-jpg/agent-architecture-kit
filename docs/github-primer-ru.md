# GitHub и репозиторий — объяснение простыми словами

## Что такое репозиторий

**Репозиторий** — это папка проекта, у которой есть история изменений.

Обычная папка:
- хранит файлы.

Репозиторий:
- хранит файлы;
- помнит, кто и что менял;
- позволяет откатываться;
- позволяет вести параллельные версии работы.

Если совсем по-простому:

**repo = проект + история изменений + правила совместной работы**

---

## Что такое Git

**Git** — это система контроля версий.

Она нужна, чтобы:
- сохранять изменения не как хаос, а как осмысленные шаги;
- видеть, что именно изменилось;
- откатывать неудачные изменения;
- вести несколько веток разработки;
- не ломать основной вариант проекта, пока экспериментируешь.

---

## Что такое GitHub

**GitHub** — это сайт/платформа, где живут git-репозитории.

Он нужен не потому, что без него Git не работает. Git и без GitHub существует.

GitHub даёт сверху:
- удобное хранение репозиториев;
- просмотр файлов в браузере;
- совместную работу;
- pull requests;
- issues;
- wiki;
- releases;
- actions/automation.

То есть:
- **Git** — это сам механизм версий;
- **GitHub** — это удобный дом и интерфейс вокруг него.

---

## Базовая терминология

### Repository / repo
Проект под Git.

### Commit
Снимок изменений.

Не просто «сохранил файл», а:
- вот набор изменений;
- вот их смысл;
- вот точка истории.

### Commit message
Короткое описание коммита.

Пример:
- `add memory contour overview`
- `split task state from memory docs`

### Branch
Ветка работы.

Например:
- `main` — основная стабильная ветка;
- `feature/memory-eval` — отдельная ветка под новую фичу.

Ветка нужна, чтобы делать изменения отдельно, не ломая основной контур.

### Merge
Слияние ветки обратно в основную.

### Pull Request / PR
Запрос на вливание изменений.

По сути это:
- «вот моя ветка»;
- «вот что я изменил»;
- «проверь и влей в main».

### Issue
Карточка проблемы, идеи или задачи.

Примеры:
- `Document task truth hierarchy`
- `Add regression examples for memory contour`

### Release
Официально отмеченная версия репозитория.

Например:
- `v0.1.0`
- `v1.0.0`

Обычно release используют, когда хотят зафиксировать стабильный публичный этап.

### Clone
Скачать репозиторий к себе на компьютер целиком.

### Push
Отправить локальные коммиты на GitHub.

### Pull
Подтянуть свежие изменения с GitHub к себе.

### Fork
Копия чужого репозитория в твоём аккаунте, чтобы менять независимо.

## Как обычно выглядит работа с GitHub

### Сценарий 1. Ты один ведёшь проект
1. Создал локальную папку.
2. Превратил её в git-репозиторий.
3. Сделал первый commit.
4. Создал repo на GitHub.
5. Запушил туда код.
6. Дальше работаешь коммитами.

### Сценарий 2. Хочешь делать аккуратно
1. `main` держишь чистым.
2. Под каждую заметную задачу создаёшь branch.
3. В branch делаешь изменения.
4. Через PR смотришь diff.
5. Потом merge в `main`.

Даже если ты один, это часто полезно.

## Что значит “собрать продуктовый репозиторий”

Это не просто «сложить файлы в папку».

Это значит:
- дать проекту понятное имя;
- определить, что именно он продаёт/показывает миру;
- выстроить структуру;
- написать README;
- отделить публичное от приватного;
- объяснить, как этим пользоваться;
- подготовить версии и roadmap.

То есть продуктовый repo — это уже не свалка артефактов, а понятный внешний объект.

---

## Как понять, что репозиторий хороший

Хороший repo обычно отвечает на 5 вопросов сразу:

1. **Что это?**
2. **Зачем это нужно?**
3. **Что внутри?**
4. **Как этим пользоваться?**
5. **Что тут уже готово, а что ещё нет?**

Если человек зашёл и за 2–3 минуты это понял — repo сильный.

## Практический минимальный состав репозитория

Обычно достаточно вот такого старта:

- `README.md` — главная страница
- `docs/` — документация
- `examples/` — примеры
- `schemas/` — схемы / форматы
- `LICENSE` — лицензия
- `.gitignore` — что не коммитить

Иногда ещё:
- `src/` — код
- `tests/` — тесты
- `scripts/` — служебные скрипты

---

## Что такое .gitignore

Это файл с перечнем того, что Git не должен тащить в репозиторий.

Туда обычно кладут:
- временные файлы;
- кеши;
- секреты;
- локальные базы;
- виртуальные окружения;
- build artifacts.

Это очень важно, чтобы случайно не запушить мусор или приватные штуки.

---

## Как работать дальше, если по-простому

Если совсем без усложнений, цикл такой:

1. меняешь файлы;
2. смотришь, что изменилось;
3. делаешь commit с нормальным названием;
4. push на GitHub;
5. периодически оформляешь release.

---

## Нормальная ментальная модель

Думай так:

- **папка** — просто место с файлами;
- **git repo** — папка с памятью об изменениях;
- **GitHub repo** — git repo, который живёт ещё и как публичная/командная поверхность.

---

## Самая частая ошибка новичка

Путать:
- «у меня есть куча файлов»
и
- «у меня есть репозиторий».

Куча файлов — ещё не repo.

Repo начинается там, где есть:
- структура;
- история;
- понятная цель;
- правила изменений.
