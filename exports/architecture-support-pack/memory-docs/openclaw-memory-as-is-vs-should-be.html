OpenClaw Memory: As-Is vs Should-Be

Слева — только то, что подтверждено или очень вероятно как живой текущий memory contour. Справа — как memory lifecycle должен выглядеть по плану и по хорошей архитектуре.

Текущее рабочее состояние: retrieval-driven memory contour с PostgreSQL и lexical backbone в центре.

Honest verdict

Сейчас память работает не как единый чистый memory brain, а как гибридный retrieval contour. Лучше всего подтверждены memory files, artifacts, handoffs и Postgres-backed lexical retrieval. Typed layer есть, но ещё не доминирует. Chats/tasks пока нельзя рисовать как полностью равноправный live retrieval layer без доп. подтверждения.

Should-Be

Целевая модель: memory lifecycle с явной дистилляцией из сырья в curated и typed memory, плюс модульный retrieval.

4. Typed-first serving + lexical fallback

Главный serving plane становится typed, а lexical retrieval остаётся сильной backing/fallback layer.

Target verdict

Идеальная модель — это не просто retrieval из файлов. Это memory lifecycle, где chats/tasks тоже являются источниками, но не в сыром виде на равных правах со всем подряд, а через нормальную extraction/distillation/promotion цепочку. После этого retrieval становится модульным, typed-first и более объяснимым.
