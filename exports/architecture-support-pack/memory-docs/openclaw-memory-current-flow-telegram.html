2. Как проходит запрос в runtime

Step 1

Приходит запрос

Пользователь спрашивает что-то, где может понадобиться память: факт, прошлое решение, контекст задачи, handoff или preference.

↓
Step 2

Request classification

Система определяет тип запроса: это recall, task-context, handoff, architecture-memory или что-то ещё.

↓
Step 3

Source planning

Решается, куда лезть за памятью: в notes, curated memory, artifacts, handoffs, PostgreSQL registry и связанные memory tables.

↓
Step 4

Lexical retrieval в Postgres

Основной рабочий мотор сейчас — lexical/full-text/trigram/heuristic retrieval, а не красивая единая typed-first memory engine.

↓
Step 5

Ranking и authority resolution

Найденные кандидаты ранжируются по релевантности, authority, freshness и типу источника. Здесь решается, чему доверять сильнее.

↓
Step 6

Serving pack

Из лучших кандидатов собирается контекст-пакет: факты, evidence, fallback, конфликтующие куски и материал для ответа.

↓
Step 7

Ответ пользователю

Модель отвечает уже не из пустоты, а опираясь на собранный memory-context.

3. Самая важная правда про текущую память

Core fact
Сейчас память работает как гибридный retrieval-контур. Это не единый «магический memory brain». Фактический мотор сейчас — Postgres + lexical search + ranking heuristics + authority logic. Typed memory core уже существует, но пока ещё не доминирует как главный serving layer.

4. Где сейчас слабое место

Problem
  • typed layer пока слабее lexical backbone
  • retrieval orchestration выглядит слишком монолитно
  • trace/debug/observability ещё недостаточно прозрачны
